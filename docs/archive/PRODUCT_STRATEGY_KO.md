## 0. TL;DR — 30초 요약

결론 먼저:

> **스프레드시트를 "핵심 View"로 유지하되, "제품의 정체성"에서 분리한다.** 제품의 정체성은 "**게임 밸런스 전용 실시간 협업 데이터베이스**"로 재정의한다. 데이터는 관계형 DB에 살고, 스프레드시트는 **여러 View 중 가장 친숙한 진입점**일 뿐이다.
>
> 시각/워크플로 DNA는 **monday.com을 "살짝" 차용**한다 — 상태 컬럼, Item Updates Feed, Automation Center, Dashboard 위젯 4가지만. 단 **Grid View는 Excel처럼 담백하게** 유지하고, monday 스타일은 Board/Dashboard View에서만 드러낸다. 역할별로 UI 밀도가 달라지는 것이 핵심.

이유 3가지:
1. **기획자는 스프레드시트가 있어야 쓴다** — 도입 실패의 99%는 "낯섦"이다
2. **결정권자는 스프레드시트만으로는 돈을 안 쓴다** — B2B 구매는 PD/디렉터가 한다. 그래서 monday.com의 상태 가시성과 Updates Feed가 필요
3. **협업은 데이터 모델에서 출발한다** — 스프레드시트 데이터 모델로는 진짜 협업이 불가능하다 (뒤에서 자세히)

이 전략은 **Airtable, Notion, monday.com, Baserow가 모두 이미 증명한 패턴**이다. 새로운 발명이 아니다. 그래서 안전하다.

## 1. 문제를 똑바로 바라보기

### 1.1 당신이 느끼는 막힘의 정체

> "스프레드시트로 쭉 밀고 나가야 할지 독립해야 할지"

이 질문은 실제로는 **두 가지 다른 질문이 섞여 있음**:

| 표면의 질문 | 진짜 질문 |
|---|---|
| UI 메타포를 바꿀까? | 제품의 **정체성(positioning)**을 어디에 둘까? |
| 스프레드시트 엔진을 버릴까? | 제품의 **핵심 가치**가 무엇인가? |
| 독립된 앱으로 갈까? | 고객이 **어느 순간에 돈을 쓰는가?** |

이 세 질문을 분리하면 답이 깨끗해진다. 하나씩 본다.

### 1.2 "협업 매우매우매우 중요" 를 해체하기

사용자는 "간편해야 함"을 반복 강조했다. 협업이 간편하려면 **네 가지 층**이 모두 갖춰져야 한다:

| 층 | 무엇 | 없으면 일어나는 일 |
|---|---|---|
| **1. 데이터 모델 층** | 누가 봐도 같은 것을 보는 구조화된 데이터 | "내 엑셀 최신판이 뭐야?" 메신저 대란 |
| **2. 동기화 층** | 실시간 다자 편집 (CRDT/OT) | 덮어쓰기 충돌, 복사본 지옥 |
| **3. 인지 층** | 프레즌스 / 커서 / 멘션 / 댓글 | "누가 왜 이렇게 바꿨어?" 추적 불가 |
| **4. 워크플로 층** | 승인 / 리뷰 / 버전 비교 / 알림 | "이거 반영해도 되나?" 결정 마비 |

**결정적 사실**: 스프레드시트 메타포만으로는 **1번 층에서 이미 막힌다**. 스프레드시트는 "자유로운 격자"라서 **데이터의 의미가 암묵적**이다. 기획자 A는 `공격력`을 정수로 기록, B는 "atk_min~atk_max" 문자열로 기록, C는 수식으로 기록. 같은 컬럼인데 타입이 다르다. 관계형 DB는 이걸 원천 차단한다.

> **"협업이 간편하다"는 데이터 모델에서 이미 80% 결정된다.**

## 2. 세 가지 경로 — 심층 분석

각 경로를 **협업 관점 중심**으로 뜯어본다.

---

### 경로 A. 스프레드시트 고수 ("Excel for games 플러그인")

**포지셔닝**: "게임 밸런싱에 특화된 고급 스프레드시트"

#### A.1 협업 시나리오 시뮬레이션
- **기획자 2인**이 같은 밸런스 시트를 동시에 편집
- 한 명이 B3 셀에 수식을 넣는 순간, 다른 한 명이 B3 셀을 덮어쓰면?
- → 셀 단위 락? 문서 단위 락? 후자는 실질적으로 2인 편집 불가

#### A.2 기술적 한계
- **Google Sheets도 내부적으로는 스프레드시트 UI + 관계형 DB 모델**이다 (그래서 협업이 된다)
- 순수 "그리드 + 수식" 모델은 구조화된 참조가 불안정:
  - 행 삭제 시 다른 시트의 참조가 깨질 수 있음
  - 컬럼 추가 시 수식 전체 재평가
  - **당신의 30초 지연 문제**가 이 모델의 한계를 이미 보여줬음 (`formulaEngine.ts:699`)

#### A.3 경쟁 환경
- Google Sheets, Excel Online: 무료 또는 M365 번들
- 당신 제품의 차별점은 "70+ 게임 수식"인데 이건 Google Apps Script로도 구현됨
- **경쟁자가 곧 같은 카테고리(Excel 플러그인 마켓)에서 $2-5 flat price로 덤빌 수 있음**

#### A.4 B2B 영업 시 부딪히는 벽
- PD: "우리 Google Workspace 쓰는데, 왜 굳이?"
- 디렉터: "버전 관리, 승인 워크플로, 감사 로그 없잖아요?"
- CFO: "Seat당 월 $50? Google Sheets는 무료인데?"

#### A.5 판결
- **달성 가능한 제품**: 월 $5-15/Seat 플러그인
- **달성 불가능한 제품**: 엔터프라이즈 ($50-200/Seat) 협업 플랫폼
- **가격 천장**: 매우 낮음. 1개 스튜디오 50 Seat × $10 = 월 $500

### 경로 B. 완전 독립 (스프레드시트 포기)

**포지셔닝**: "노드 기반 게임 밸런스 시뮬레이터" 또는 "게임 디자인 BI"

#### B.1 협업 시나리오
- 노드 그래프 기반 편집 (Machinations처럼)
- 다중 사용자가 노드를 드래그/연결하는 Figma-style 멀티플레이
- **기술적으로 가능**: Figma, Miro가 CRDT 기반 협업 그래프를 구현함

#### B.2 문제점
- **기획자가 떠남**: 게임 기획자의 근력은 Excel이다. 노드 그래프 러닝커브는 심리적 장벽
- **입력 효율 급락**: 100개 몬스터 스탯을 노드 그래프로 입력? 엑셀이면 10분, 노드면 몇 시간
- **Machinations가 이미 있음**: 경쟁사가 10년 선점, 유료 SaaS로 자리잡음

#### B.3 시장 증거
[Machinations.io](https://machinations.io/)는 2026년에도 여전히 "소수의 시스템 디자이너용" 도구로 남아있다. 대중화 안 됨. 이유는 **엑셀을 이기지 못했기 때문**.

#### B.4 판결
- **달성 가능**: 기존 경쟁사(Machinations, PlayFab)와 정면 대결
- **치명적 리스크**: 이미 있는 싸움에 뒤늦게 합류
- **가격 천장**: 중간 ($30-80/Seat)이지만 진입 장벽도 높음

### 경로 C. 하이브리드 ("게임 밸런스 전용 Airtable + LiveOps") ⭐추천

**포지셔닝**: "**스프레드시트처럼 편하고 데이터베이스처럼 강한 게임 밸런스 협업 플랫폼**"

#### C.1 데이터 모델 (핵심)

```
[ PostgreSQL (또는 PG + TimescaleDB) ]
         │
         ├── 테이블: Characters  (id, name, tier, hp, atk, ...)
         ├── 테이블: Weapons     (id, name, damage, crit, ...)
         ├── 테이블: Stages      (id, wave, enemy_id→)
         └── 테이블: Formulas    (정의된 커스텀 수식)
              │
              └─── 관계(FK) 자동 유지

   ▲ 데이터는 여기에만 "산다"                   ▲ 데이터 무결성 보장
   ▼ UI는 여러 View로 투영                     ▼ 협업/버전 관리/감사 여기서
```

#### C.2 View 계층 (사용자가 보는 것)

같은 데이터를 **목적별로 다른 방식으로** 보여준다:

| View                  | 주 사용자    | 용도                                |
| --------------------- | -------- | --------------------------------- |
| **Grid** (스프레드시트 UI)  | 기획자      | 대량 편집, 수식, 복붙 — **기존 사용자 근력 그대로** |
| **Kanban**            | PD       | 밸런스 변경 티켓 상태 추적                   |
| **Gantt**             | PM       | 패치 일정, 밸런스 롤아웃                    |
| **Dashboard**         | 디렉터      | KPI, 스냅샷 통계 트렌드                   |
| **Diff Viewer**       | 리뷰어      | 스냅샷 간 변경점 (Git-style)             |
| **Simulation Report** | 밸런서      | Monte Carlo 결과, 승률 히트맵            |
| **Graph** (노드)        | 시스템 디자이너 | 경제 Faucet/Sink, Perfect Imbalance |

**핵심**: 같은 `Characters` 테이블을 Grid에서 수정하면 Kanban, Dashboard가 실시간 반영된다. 복사본이 존재하지 않는다.

#### C.3 협업이 "매우매우매우 좋은" 이유

| 협업 층        | 구현 방식                                      | 사용자 체감                                     |
| ----------- | ------------------------------------------ | ------------------------------------------ |
| **데이터 모델**  | 관계형 DB + 스키마 검증                            | "데이터 형식 안 맞아요" 경고가 입력 시점에 뜸                |
| **실시간 동기화** | **Yjs CRDT** + y-websocket + Redis Pub/Sub | Figma처럼 다른 사람 커서가 보이고, 내가 타이핑하는 순간 모두에게 반영 |
| **프레즌스**    | 각 View에 접속한 유저 표시                          | 누가 지금 어느 시트를 보고 있는지 항상 보임                  |
| **댓글/멘션**   | 셀/행 단위 댓글 스레드, @멘션 알림                      | monday.com처럼 맥락에서 바로 토론                    |
| **버전/감사**   | 모든 변경을 스냅샷 + 이벤트로 기록                       | "누가 언제 뭘 왜 바꿨나" 1초 조회                      |
| **워크플로**    | 변경 → 리뷰 요청 → 승인 → 머지                       | Perforce Swarm + Git PR 모델을 밸런스에 적용        |
| **Git 연동**  | Webhook 양방향 (push → 시트, 시트 → PR)           | 개발자는 코드 레포에, 기획자는 UI에 — 자동 싱크              |

#### C.4 기술 스택 (검증된 조합)

- **실시간 엔진**: [Yjs](https://github.com/yjs/yjs) — Notion, Linear, GitLab이 쓰는 CRDT. 오프라인 편집 + 자동 머지 + 무한 확장
- **전송**: y-websocket 또는 [y/hub](https://github.com/yjs/yhub) + Redis Streams (수평 확장)
- **영구 저장**: PostgreSQL에 Yjs 문서 바이너리 저장 (기존 DB와 자연 공존)
- **인증**: Keycloak (SAML/LDAP/SSO — 엔터프라이즈 필수)
- **스냅샷/통계**: TimescaleDB Continuous Aggregate
- **오픈소스 레퍼런스**: [Baserow](https://baserow.io/), [NocoDB](https://github.com/nocodb/nocodb) — 둘 다 이 모델의 성공을 증명

#### C.5 이 모델이 이미 승리한 증거

| 제품 | 모델 | 규모 |
|---|---|---|
| **Airtable** | 관계형 DB + Grid/Kanban/Calendar/Gantt View | Fortune 100의 80%, 50만+ 조직 |
| **Notion** | 블록 DB + 다중 View | 1억+ 사용자 |
| **monday.com** | Board + Multiple Views + Updates | 20만+ 고객, NASDAQ 상장 |
| **Linear** | Issue DB + List/Board/Timeline | 개발자 표준 |
| **Baserow** (오픈소스, 자체호스팅) | Airtable 대체 | GDPR/HIPAA/SOC2, 15만+ 사용자 |

**이 5개 제품이 모두 "데이터베이스 + 다중 뷰 + 실시간 협업" 패턴을 공유한다.** 이것은 트렌드가 아니라 **B2B 협업 도구의 표준 아키텍처**이다.

#### C.6 판결
- **달성 가능**: 월 $50-150/Seat 엔터프라이즈 협업 플랫폼
- **리스크**: 중간 (기술 복잡도 O, 하지만 검증된 패턴)
- **가격 천장**: 높음. 100 Seat 스튜디오 × $100 = 월 $10,000

## 3. 왜 하이브리드가 "간편함"도 이기는가

당신은 "간편해야 한다"를 강조했다. 직관적으로는 스프레드시트가 가장 간편해 보인다. **하지만 이건 틀렸다.**

### 3.1 "간편함"의 진짜 정의

**간편함 = 학습 비용이 낮다 × 반복 작업 비용이 낮다 × 실수 복구 비용이 낮다**

스프레드시트는 첫 항은 잘하지만, 나머지 둘은 오히려 약하다:

| 시나리오 | 순수 스프레드시트 | 하이브리드 |
|---|---|---|
| "캐릭터 50명 티어별 정렬" | 수동 정렬 또는 필터 | 1초 (DB 쿼리) |
| "이 컬럼이 숫자여야 함" 검증 | 사후 VBA/Script | 스키마에서 즉시 차단 |
| "변경 전으로 되돌리기" | Ctrl+Z 한계 존재 | 스냅샷/버전에서 선택 복원 |
| "누가 이 값을 바꿨어?" | 변경 이력 없음 | 감사 로그에서 즉시 조회 |
| "이 캐릭터 참조하는 스킬들" | VLOOKUP + 검색 | 관계 자동 탐색 (1클릭) |
| "밸런스 변경을 검토 후 반영" | 이메일/메신저 | 리뷰 요청 → 승인 워크플로 |

**간편함은 UI의 친숙함이 아니라 "일을 끝내기까지의 총 마찰"이다.** 하이브리드가 UI 친숙함은 같거나(Grid 뷰 덕분) 약간 떨어지지만, 나머지 모든 축에서 압도한다.

### 3.2 "스프레드시트 내는 그대로" + "백그라운드에서 데이터베이스"

하이브리드의 교묘함: **사용자는 자기가 데이터베이스를 쓰고 있는 줄 모른다.**

- Grid View는 Excel처럼 동작
- 셀 편집은 Excel처럼 동작
- 수식 입력도 `=DPS(A, B)` 그대로
- **하지만 내부는 타입이 있는 필드, 관계가 있는 테이블, 실시간 동기화되는 CRDT**

이게 Airtable이 10년간 검증한 요리법이다.

## 4. 협업 기능 상세 설계 (가장 중요한 부분)

사용자가 "매우매우매우"로 강조한 부분. 실제로 어떻게 구현되는지 구체적으로.

### 4.1 실시간 동시 편집 — Figma/Notion 수준

**기술**: Yjs CRDT + y-websocket

**사용자 경험**:
```
┌─────────────────────────────────────────────────┐
│ 🟢 김기획 (B3 편집 중)                            │
│ 🟡 박디렉터 (viewing Dashboard)                  │
│ 🔵 이밸런서 (C7 편집 중)                          │
├─────┬────────┬────────┬────────┬─────────────────┤
│     │  이름  │   HP   │   ATK  │  TIER           │
│ 1   │ 전사   │  1200  │   85   │   A             │
│ 2   │ 마법사 │   800  │  120   │   S             │
│ 3   │ 도적  │[🟢150] │   95  │   B              │ ← 김기획의 커서
│ 4   │ 힐러   │   900  │   45   │   A             │
│ 5   │ 탱커   │  1500  │   60   │  [💬 2 comments]│ ← 댓글 인디케이터
│ ... │                                            │
│ 7   │ 사제   │   750  │  [🔵] │   B              │ ← 이밸런서의 커서
└─────┴────────┴────────┴────────┴─────────────────┘
```

**동작 원리**:
- 모든 셀 편집은 Yjs CRDT operation으로 변환
- WebSocket으로 서버 전파 → 모든 클라이언트 동기화
- 충돌은 수학적으로 불가능 (CRDT의 수렴성 보장)
- 오프라인 편집도 지원 — 재연결 시 자동 머지

### 4.2 프레즌스 (누가 어디 있는지 보이기)

- 사이드바: 현재 접속한 팀원 아바타
- 각 View 탭: 지금 이 View를 보고 있는 사람 아이콘
- 각 행/셀: 지금 편집 중인 사람의 커서
- 전체 프로젝트: "이번 주 활동 히트맵"

### 4.3 댓글 & 멘션

**셀/행 단위 스레드**:
- 셀에 우클릭 → "댓글 달기"
- `@박디렉터 이 수치 이대로 가도 되나요?`
- @멘션 받은 사람은 즉시 알림 (앱 내 + 이메일 + Slack Webhook)
- 댓글 해결(Resolve) 가능 — 해결되면 인디케이터 사라짐

**문맥 연결**:
- 댓글이 셀에 붙어 있음 → 1주 뒤에 봐도 맥락 사라지지 않음
- monday.com의 "Communication happens ON the work item" 철학 그대로

### 4.4 변경 이력 & 감사 로그

**모든 변경을 기록**:
```
2026-04-18 14:32  김기획  Characters.도적.HP: 120 → 150
                  이유: "레벨 20 테스트 플레이 피드백"
                  연결: [Git commit abc123] [Task PB-142]

2026-04-18 14:35  박디렉터  Characters.도적  [댓글]
                  "다음 패치에 반영하죠"

2026-04-18 15:10  이밸런서  Characters.도적.HP: 150 → 145
                  이유: "Monte Carlo 시뮬 결과 생존율 +8% 과도"
                  시뮬 링크: [Simulation #2891]
```

이게 그냥 "로그"가 아니라 **제품 자체**다. 이걸 타임라인 뷰로 스크롤하면서 "이 밸런스 변경의 서사"를 읽을 수 있다.

### 4.5 리뷰 & 승인 워크플로

**Perforce Swarm + GitHub PR 스타일**:
1. 기획자가 브랜치(Draft)에서 변경
2. "리뷰 요청" 버튼 → PD 지정
3. PD는 Diff View에서 변경점만 확인
4. 댓글로 논의, 추가 수정 요청
5. 승인 → main에 머지 → 실제 게임 반영

**이것이 B2B의 킬러 기능**: 기획자 혼자 프로덕션 밸런스를 바꾸지 못한다. 감사/컴플라이언스 통과.

### 4.6 Git 양방향 연동

**시트 → Git**: 기획자가 편집 → 머지 시 자동으로 `balance.json` 커밋 + PR 생성

**Git → 시트**: 개발자가 `balance.json` 직접 수정 → push 시 Webhook으로 시트 업데이트 + 팀에게 알림

**결과**: 기획자와 개발자가 **서로의 도구를 바꿀 필요 없이** 같은 데이터를 조작.

### 4.7 알림 (간편함의 마지막 마일)

monday.com 스타일 세밀한 필터:
- [ ] 나한테 멘션됐을 때
- [ ] 내가 구독한 시트가 변경됐을 때
- [ ] 내 리뷰가 요청됐을 때
- [ ] 자동화 규칙이 트리거됐을 때
- 채널: 앱 내 / 이메일 / Slack / 카카오워크 / 모바일 푸시

### 4.8 monday.com 스타일 — "살짝" 가져오는 요소들

monday.com 전체를 복제하지 않는다. **결정권자(PD/디렉터)에게 효과가 큰 4가지만** 선택적으로 가져오고, 기획자의 편집 흐름은 건드리지 않는다. 원칙은 하나:

> **"Grid View는 Excel처럼 담백하게, Board/Dashboard View에서만 monday.com의 시각적 경쾌함"**

#### 4.8.1 Status 컬럼 (색상 기반 상태)

특수한 컬럼 타입 `status`를 추가한다. 일반 텍스트/숫자 컬럼과 달리 **정해진 옵션 + 색상**만 들어간다.

예시:
```
┌──────────┬─────────┬────────────────────────┐
│  이름    │   HP    │   상태                 │
├──────────┼─────────┼────────────────────────┤
│  전사    │   1200  │  🟢 승인됨             │
│  마법사  │    800  │  🟡 검토 중            │
│  도적    │    150  │  🔴 롤백 예정         │
│  힐러    │    900  │  🔵 Draft              │
│  탱커    │   1500  │  ⚫ 보류               │
└──────────┴─────────┴────────────────────────┘
```

구현:
- DB 스키마: `{ type: 'status', options: [{id, label, color}] }`
- UI: 셀 클릭 → 옵션 드롭다운 (색상 팔레트 포함)
- 필터/정렬에 바로 반영 ("승인됨만 보기")
- Board View에서 **열(column) 축으로 자동 전환** — 칸반이 됨
- Dashboard에서 상태별 개수 파이차트로 자동 집계

왜 살짝만:
- Grid View에서 상태 컬럼은 **옵션**이지 필수 아님. 기획자가 원하지 않으면 안 써도 됨
- 상태 옵션은 **보드별로 커스터마이즈** 가능 (밸런스 보드와 버그 보드가 다른 상태를 가짐)

#### 4.8.2 Item Updates Feed (행 단위 소통 스레드)

monday.com의 **시그니처 기능**. 각 행(Item)에 자체 업데이트 피드가 붙는다.

기존 §4.3 댓글은 "셀에 한 줄 스레드". Updates Feed는 "행 전체의 활동 타임라인":

```
┌──────────────────────────────────────────────────┐
│ 도적 (Character)                        [닫기 ✕] │
├──────────────────────────────────────────────────┤
│ 📊 HP: 150 | ATK: 95 | TIER: B                   │
├──────────────────────────────────────────────────┤
│ 💬 Updates                            [+ 업데이트]│
│                                                   │
│  박디렉터 · 2시간 전                              │
│  "다음 패치에서 HP 조정하죠. @김기획 시뮬 부탁"    │
│  👍 2  💬 답글 3                                  │
│                                                   │
│  김기획 · 1시간 전                                │
│  Monte Carlo 시뮬 결과 첨부:                      │
│  📎 sim_results_2026-04-18.pdf                    │
│  "생존율 +8% 나와서 145가 적절해 보입니다"        │
│                                                   │
│  [자동] · 10분 전                                 │
│  ✏️ HP: 150 → 145 (by 이밸런서)                   │
│  🔗 Git commit: abc123                            │
└──────────────────────────────────────────────────┘
```

차이점:
- 댓글(§4.3)은 **셀 단위 마이크로 스레드**
- Updates Feed는 **행 단위 타임라인** — 자동 변경 이벤트도 같이 섞임
- 파일 첨부, 이모지 반응, 답글, 해결 마크 모두 지원
- **활동 로그(§4.4)와 자동 합쳐짐** — 별도 탭 아닌 한 흐름

왜 살짝만:
- Grid View에서는 해당 행에 **댓글 개수 뱃지만** 표시 (`💬 3`)
- 클릭해야 사이드 패널로 열림 — **평소엔 안 보임** = Grid 집중력 해치지 않음

#### 4.8.3 Automation Center (no-code 규칙)

"이 조건이면 → 저 동작" 규칙을 UI로 만든다. 코드 한 줄 없이.

템플릿:

```
WHEN 상태가 "승인됨"으로 변경되면
THEN 
  1. Slack #balance-channel에 알림
  2. Git에 자동 커밋
  3. @개발팀에 멘션
  4. 스냅샷 자동 생성

WHEN 시뮬레이션 결과 승률 차이 >15% 이면
THEN
  1. 상태를 "⚠️ 불균형 감지"로 자동 변경
  2. 담당자에게 이메일
  3. 해당 행에 자동 댓글 추가

WHEN 매주 금요일 17:00
THEN
  1. 이번 주 밸런스 변경 요약을 디렉터에게 전송
  2. 스냅샷 자동 생성하여 주간 빌드에 포함
```

구현:
- Trigger × Condition × Action 3단 DSL
- JSON으로 저장, 서버 워커가 실행
- 초기 MVP는 **10개 레시피 템플릿**만 제공 (사용자가 파라미터만 수정)

왜 살짝만:
- monday.com은 100+ 자동화 레시피. 우리는 **게임 밸런스 워크플로 특화 10-20개**만
- Zapier/n8n처럼 복잡한 분기는 버림 — "간편함" 지키기 위해

#### 4.8.4 Dashboard 위젯 (디렉터 전용 집계 View)

Grid/Board와 독립된 **읽기 전용 대시보드**. 디렉터/PD가 매일 아침 열어서 1분 만에 상황 파악.

위젯 종류:
- 📊 **상태 분포** — 각 상태별 Item 개수 (파이/막대)
- 📈 **트렌드** — "지난 30일 밸런스 변경 추이" (TimescaleDB 활용)
- 🗓️ **이번 주 일정** — Gantt에서 자동 추출
- 🔥 **핫 이슈** — 댓글 많은 Item Top 5
- 🧪 **시뮬 대시보드** — 최근 시뮬레이션 승률 히트맵
- 👥 **팀 활동** — 기획자별 편집 횟수 히트맵
- 📉 **KPI** — 사용자가 지정한 커스텀 수치 (평균 DPS, 총 재화 유입 등)

위젯은 **드래그로 재배치**, 팀 단위로 저장 가능. 초기엔 5-7개 프리셋만.

### 4.9 밀도(Density) 모드 — "모드별로 UI가 차분해지거나 풍부해진다"

monday.com 엘리먼트를 넣되 기획자를 방해하지 않는 비밀은 **View마다 시각적 밀도가 다르다**는 원칙:

| View | 색상 사용 | 애니메이션 | 폰트 | 목표 |
|---|---|---|---|---|
| **Grid** | 최소 (상태 컬럼만) | 없음 (CRDT 업데이트 하이라이트만) | 고정폭 선호 | **편집 집중, 조용함** |
| **Kanban/Board** | 풍부 (카드 색상, 상태 배지) | 드래그 애니메이션 | 비례폭, 조금 큼 | **상태 한눈에** |
| **Dashboard** | 풍부 (차트 팔레트) | 숫자 카운트업, 차트 전환 | 헤드라인 강조 | **발표/리뷰용** |
| **Timeline/Gantt** | 중간 (프로젝트별 색상) | 스크롤 시 부드러운 줌 | 일반 | **일정 개괄** |

즉, 같은 제품 안에서 "**조용한 Excel 모드**"와 "**경쾌한 monday 모드**"가 공존한다. 사용자는 자신의 역할에 맞는 View에서 주로 시간을 보내면 됨.

## 5. "간편함"을 지키기 위한 UX 원칙

사용자가 "설명문서 없으면 뭐가 어디 있는지 모름"을 지적했다. 해결책:

### 5.1 Command Palette (`⌘K`)
모든 기능을 검색으로 접근. Linear, Notion, Figma가 이미 표준화한 패턴.

### 5.2 Contextual Help
모든 도구 패널 우측 상단 `?` → 인라인 설명 + 30초 스크린캐스트. 별도 문서 사이트 불필요.

### 5.3 Empty State 스토리텔링
빈 프로젝트를 열면 "시작하려면: ① 샘플 불러오기 / ② 템플릿 선택 / ③ Git 연동" 3개 카드.

### 5.4 역할별 투어
기존 `useTour` 훅을 확장해 **"기획자 모드 / PD 모드 / 개발자 모드"** 각각 다른 온보딩.

### 5.5 인라인 AI 도우미 (선택)
"어떻게 DPS 계산해?" → 사내 LLM이 해당 도구로 딥링크 + 스텝 가이드. 설명서 0페이지로 운영 가능.

## 7. 의사결정 프레임워크

### 7.1 "5년 뒤 당신이 원하는 모습"으로 역산

- **"게임 기획자들이 내 플러그인 쓰면서 편해졌다"**라면 → 경로 A (스프레드시트)
- **"한국 게임사 10곳이 내 플랫폼으로 밸런스 관리한다"**라면 → 경로 C (하이브리드)
- **"게임 밸런스 시뮬레이션의 표준이 됐다"**라면 → 경로 B (완전 독립, Machinations와 정면)

### 7.2 "자본/시간 투자 대비 회수"

| 경로 | 개발 기간 | 목표 ARR (5년) |
|---|---|---|
| A | 3-6개월 | $50K-200K |
| B | 12-18개월 | $500K-2M (성공 시) / $0 (실패 시) |
| C | 9-12개월 MVP | $300K-3M |

### 7.3 "협업 요구를 충족할 수 있는가"

## 11. 반복 스트레스 테스트 과정

5개 클러스터(고객/기술/시장/실행/제품) 각각에 대해 방어-공격 사이클을 돌렸다. 각 라운드는 "방어(해결안) → 공격(약점 지적)"으로 구성.

### 11.1 클러스터 A: 고객 검증 & 타겟 시장

**원본 문제**: 고객 대화 0건으로 쓴 전략. 한국 중견 게임사 = B2B 지옥. 1인 9개월 MVP 망상.

**R1 방어**
- 4주 고객 발굴 인터뷰 10회
- Stripe 유료 버튼 테스트로 지불 의향 검증
- 타겟을 인디/소규모로 하향
- MVP 최소화: 30초 지연 + 기본 협업만

**R1 공격**
- 콜드 아웃리치 응답률 3-5%, 10회 대화 확보에 200-300 메일 필요
- "무엇이 가장 고통인가?" → 추상적 답변만 반환, 실질적 통찰 없음
- 인디는 유료 전환율 0.5-1%, $9/월로 생존 어려움
- "MVP 최소화" 모호함 — 협업 하나만 넣어도 auth/백엔드 필수

**R2 방어**
- 콜드 아웃리치 금지. 기존 채널 활용: Vercel Analytics + GitHub 스타저 DM
- 포괄적 질문 금지. 프로토타입 3개 + Stripe 결제 의향 측정
- 실시간 협업 포기. **Git 기반 비동기 협업**(diff, history, blame)으로 진짜 고통("누가 최신판이야?") 해결

**R2 공격**
- Vercel Analytics에 이메일 없음, 직접 접촉 경로 없음
- GitHub 스타저 DM은 스팸 취급
- 프로토타입 3개 = 3배 공수, 2개는 버리게 됨
- Git 비동기 협업은 GitKraken/GitHub Web이 이미 제공 — 차별화 약함
- 한국 게임 개발자는 GitHub보다 **Inven/네이버카페/디스코드** 체류

**R3 수렴**
- **채널 전략**: 한국 게임 개발자 커뮤니티(Inven 게임개발 게시판, `게임넥스트` 카페, Discord 게임 개발 서버)에 "무료 체험 + 피드백 교환" 포스팅. 상호 호혜 명분. 자발적 응답만 받음.
- **질문 설계**: 추상적 질문 대신 "이거 $5/월이면 쓰시겠어요?" 구체 질문 + Google Form 금액 슬라이더
- **최소 검증 제품**: 현재 Balruno 유지 + **딱 1개 유료 기능 추가** — "**Git Backup + Version History**". GitHub API를 백엔드로 활용해 자체 서버 구축 회피
- **가격**: $5/월 (마찰 최소)
- **성공 기준**: 3개월 내 유료 10명
- **포기 기준**: 4주 인터뷰 응답 0 OR 3개월 유료 전환 0 → 사업 포기 또는 대안 경로(라이프스타일 OSS) 전환

**잔존 위험 (수용)**
- 한국 인디 개발자 지불력이 스튜디오와 다를 수 있음 — Phase 1에서 밝혀짐
- GitHub 의존 백엔드는 향후 온프렘 요구와 불일치 — Phase 3에서 재설계 필요
- Toss Payments 등 국내 결제 연동 복잡성 존재

### 11.2 클러스터 B: 기술 설계 (Yjs / TimescaleDB / Perforce)

**원본 문제**: Yjs를 스프레드시트에 쉽게 말한 오판. TimescaleDB 과투자. Perforce 미대응.

**R1 방어**
- Yjs 제거. 시트 전체를 스냅샷으로 저장, last-write-wins
- TimescaleDB 제거, PostgreSQL + 야간 집계
- Perforce는 나중 플러그인

**R1 공격**
- Last-write-wins = 데이터 유실, B2B에서 치명적
- 단순 버전 관리 = 2005년 Google Docs 수준, 사용자 실망
- Perforce 플러그인도 자체로 프로젝트 1개 분량

**R2 방어**
- 행 단위 낙관적 락: 편집 진입 시 "체크아웃", 다른 유저는 "editing..." 표시 후 충돌 블록
- PostgreSQL만 영구 사용. 분석 필요 시 ClickHouse 추가 (선제 최적화 X)
- Perforce: **V1에서 명시적 미지원**. GitHub/GitLab 샵만 공략. 수익 발생 후 외주로 Perforce 어댑터 구축 ($20K 1회)

**R2 공격**
- 행 락 = "Figma for balance" 비전 사망 — 동시 편집 불가
- "GitHub/GitLab만" 한정 시 한국 시장 거의 없음 (대부분 Perforce)
- Perforce 외주 = 수익 전제. 수익 없으면 불가 (순환)

**R3 수렴**
- **비전 재정의 (중요)**: V1은 "Figma for balance" **아님**. "**Airtable for game balance with Git backup**"으로 솔직히 다운스코프. 실시간 멀티플레이어는 V3+ 이후.
- **기술 스택 최소화**:
  - 프론트: Next.js 16 그대로
  - 백엔드: **Node.js/TypeScript + Hono/Fastify** (Rust 학습 비용 과다)
  - DB: PostgreSQL, JSONB로 시트 저장
  - 인증: **GitHub OAuth만** (Keycloak 없음, 비용 0)
  - 컨테이너: 단일 Docker 이미지 (Compose도 초기엔 불필요)
- **온프렘**: 첫 유료 B2B 고객 요청 시에만 구축. **주문 개발 계약으로 비용 회수**. Day 1에 온프렘 준비 X
- **Perforce**: V1-V2 미지원. .csv Import/Export로 우회 가능. V3 이후 외주 검토

**잔존 위험**
- 실시간 협업 없으면 Airtable 대비 차별화 약함
- 온프렘 스토리 초기 부재 = 엔터프라이즈 리드 일부 상실
- 기술 부채: 단순히 시작하면 나중 확장 시 리팩터 필요

### 11.3 클러스터 C: 시장 규모 & 가격 & Moat

**원본 문제**: TAM 계산 없음. $50-150/Seat 근거 없음. "전체 조합 = moat" 허구.

**R1 방어**
- TAM 정직 계산, 가격 A/B 테스트, 도메인 전문성을 moat로

**R1 공격**
- 고객 데이터 없이 TAM = garbage in garbage out
- 트래픽 부족 시 A/B 통계 불가
- 도메인 전문성 moat 아님 — GDC 영상 본 사람 누구나 카피

**R2 방어**
- 보수적 TAM: 글로벌 스튜디오 2.5만 → 중견+ 2.5K → 외부 툴 사용 125 → 온프렘 허용 25 → 1년차 포획 2-5
- 5년차 낙관 $500K-1.5M ARR
- Moat 없음 공식 인정

**R2 공격**
- 2-5 고객도 낙관적
- "Moat 없음, 속도로 경쟁" = 큰 경쟁자 등장 시 즉시 증발

**R3 수렴**
- **사업 규모 재정의**: 이것은 **라이프스타일 비즈니스**, 벤처 아님. 기대 매출 지표를 그에 맞춤
  - Year 1: MRR $1,000 (50 유저 × $20)
  - Year 2: MRR $5,000
  - Year 3: MRR $20,000 → **결정 지점**: 라이프스타일 유지 vs 스케일업(공창/투자)
- **가격 사다리**:
  - Free: 인디 제품 영원히 무료 유지
  - Pro $9/월: Git Backup + Version History (단일 기능)
  - Team $29/유저/월: 공유 워크스페이스 (트래픽 후)
  - Enterprise: inbound 딜만
- **Moat 전략**: 구조적 moat 없음 공식 수용. 대신 **중력(gravity)**:
  - 진짜 훌륭한 무료 티어
  - 게임 밸런스 이론 커뮤니티 콘텐츠
  - 빠른 반복
  - 사랑받는 제품 (Linear 플레이북)

**잔존 위험**
- Year 1 $1K MRR 자체가 불확실 ($0일 수 있음)
- "Moat 없음" 수용 = 경쟁자 위협 상시 존재
- "라이프스타일 비즈니스" 프레임이 사용자의 실제 야망과 불일치 가능

### 11.4 클러스터 D: 실행 (1인 타당성)

**원본 문제**: 1인이 풀스택 + 세일즈 + 지원 = 1년 내 번아웃. 최근 커밋 궤적은 인디 OSS지 엔터프라이즈 아님.

**R1 방어**
- 공동창업자 모집, VA 고용, 커뮤니티 마케팅

**R1 공격**
- 공창 탐색 6-12개월 → 런웨이 없이 불가
- VA 고용도 수익 필요 (순환)
- B2B 커뮤니티 마케팅 전환율 낮음

**R2 방어**
- B2B 포기, Pro 티어에 집중 (1인 감당 가능)
- 지원은 비동기 전용 (이메일, Intercom), 통화 없음
- 월 1회 심층 블로그로 SEO 장기 축적

**R2 공격**
- B2B 꿈 포기 = 원래 논의 뒤집기
- SEO 2-3년 지연, 그 사이 수입 저조
- 주 20시간도 부업 기준 많음

**R3 수렴**
- **단계적 야망 (결정적 원칙)**:
  - **Phase 1 (0-6월)**: Pro 티어만. 지불 의향 검증. 인디 OSS는 영원 무료 유지.
  - **Phase 2 (6-18월)**: MRR >$3K 달성 시 Team 티어 탐색.
  - **Phase 3 (18월+)**: MRR >$10K + 복수 조직 검증 시에만 B2B 검토. **이때도 revenue가 운영비 감당하는 경우에만**.
  - **B2B/엔터프라이즈 = 하위 티어 트랙션으로 얻어내는 것, Day 1 기획 아님**
- **명시적 중단 조건**: Phase 1 끝에 MRR <$500 이면 B2B 꿈 완전 포기. 더 이상 시간 투입 X
- **워라밸**: 최대 주 25-30시간. 본업/컨설팅 병행으로 안정 수입 확보. Balruno는 입증 전까지 사이드 프로젝트
- **3개월 결정 게이트**: MRR 목표 미달 → 스코프 축소. 달성 → 확장

**잔존 위험**
- 단계적 접근 = 시장 기회 창 대비 느릴 수 있음
- 본업 + 사이드 = 둘 다 품질 저하 가능
- B2B 단계 영원히 도달 못 할 수 있음 — 수용 필요

### 11.5 클러스터 E: 제품 방향 모순

**원본 문제**: 협업 수요 검증 안 됨. "살짝 monday"가 실은 "전부 monday". 게임 함수 moat 주장 모순.

**R1 방어**
- V1에서 실시간 협업 제외, monday 요소 제외, 단일 유저 Pro 기능에 집중

**R1 공격**
- 그러면 Excel 대비 차별화 무엇? B2B 논리 붕괴
- 원래 거부했던 경로 A(엑셀 플러그인)로 퇴보

**R2 방어**
- V1 협업 재정의: **비동기** — Git 커밋, 행 댓글, @멘션 via GitHub Issues
- 실시간은 후에 검증 시 추가
- 상태 컬럼 딱 1개만 추가 (monday 요소 최소화)
- 게임 함수를 moat 아닌 **생산성 도구**로 재포지셔닝

**R2 공격**
- "Git 비동기 협업"은 GitHub가 이미 함. 왜 돈 내나?
- 상태 컬럼 1개는 "살짝 monday" 라벨 붙일 만큼 큰 변화 아님
- "생산성 차별화" 여전히 약한 포지셔닝

**R3 수렴**
- **포지셔닝 완전 재설정**: "**Excel이 아쉬울 때 게임 디자이너가 집어드는 도구**"
  - "Figma for balance" 포기
  - "Airtable for game balance" 포기
  - 그냥: "Excel이 한계 느껴질 때 자연스럽게 사용하는 게임 밸런스 전용 도구"
- **협업 최소 형태**:
  - Team 티어에서 공유 워크스페이스
  - 행 단위 댓글 (비동기)
  - Git 기반 히스토리 (누가 무엇을 언제)
  - **실시간 멀티플레이어 커서는 V1-V2 제외**
- **monday 요소**:
  - 솔직히: "monday.com에서 상태 컬럼 아이디어 차용"
  - 상태 컬럼 하나만
  - Automation/Dashboard 위젯 V1 제외
  - "살짝 monday" 과장 표현 자제
- **게임 함수 포지셔닝**:
  - moat 아님 공식 철회
  - 포지션: **"내장 게임 공식 40+로 기획자 주당 2-3시간 절약"** — 구체 생산성 주장, 측정 가능

**잔존 위험**
- "Figma for balance" 흥분 상실 — 얼리 팬층 일부 실망
- 작은 비전이 얼리 어답터에게 언더웰밍
- "Excel보다 나은"이 구매 트리거로 충분한지 여전히 불확실

## 14. 수정 후에도 남는 위험 (Residual Risks)

완벽한 계획은 없다. 위 수정안으로도 다음은 남는다:

1. **한국 인디 게임 개발자의 지불 의사가 실제로 낮을 가능성** — Phase 1에서 밝혀짐. 대응: 글로벌 영어권 타겟 전환 고려
2. **GitHub API 의존 백엔드가 후에 온프렘 수요와 충돌** — Phase 3에서 재설계 필요
3. **6개월 MRR $500도 낙관적 가능성** — 0일 수도 있음. 냉정 수용
4. **"Better Excel" 포지셔닝이 불충분한 매력** — 시장 응답으로만 확인 가능
5. **Airtable/Notion이 게임 템플릿 출시 시 즉시 증발** — moat 부재 수용
6. **한국 게임 커뮤니티에서 "무료 제공 + 피드백" 제안도 무반응일 가능성** — 대안: Reddit /r/gamedev, itch.io 게임 제작자 커뮤니티
7. **Stripe/Toss Payments 결제 연동 공수 과소평가** — 주말 1회 작업 가정했지만 실제 1-2주 소요 가능
8. **본업/컨설팅과 사이드 병행의 실제 지속 가능성** — 6개월 후 번아웃 전조 시 즉시 중단

이 위험들은 수정안이 **덜 야심차지만 더 생존 가능한** 계획임을 의미한다. 야망보다 **생존**을 우선시한 것이 반복의 핵심 결과.
